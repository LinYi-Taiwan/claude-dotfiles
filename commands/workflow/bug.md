你是一位資深工程師，請處理以下問題：

**問題描述：** $ARGUMENTS

---

## Step 1 - 判斷情境

先判斷屬於哪種情況，然後直接執行對應流程：

- **無法啟動 / 一直報錯** → 執行「Loop 修復流程」
- **知道哪裡壞** → 執行「快速修復流程」
- **不知道原因** → 執行「調查後修復流程」

---

## Loop 修復流程（無法啟動）

不斷執行以下循環直到成功為止，**不要問我，自己決定**：

1. 嘗試啟動（查 `package.json` / `README` 找啟動指令）
2. 成功 → 結束
3. 失敗 → 找最根本的錯誤（看最底層的 error）
4. 修復（套件問題裝套件、設定問題補設定、程式碼問題改程式碼）
5. 回到 1

**原則：**

- 每次只修一個問題
- 同一個錯誤出現 3 次換方法
- 超過 10 個循環沒解決才來報告

---

## 快速修復流程（原因明確）

### 1. 確認問題

- 重現 bug：找到觸發條件，確認能穩定重現
- 如果有錯誤訊息，先讀完整 stack trace 找到出錯的 code 位置
- 讀相關 code，確認自己理解問題的根因

### 2. 評估範圍

- 這段 code 被多少地方使用？改了會影響誰？
- 有沒有相關的測試？測試涵蓋到這個情境嗎？
- 列出要改的檔案清單（預計不超過 3 個檔案，超過要重新評估是不是快速修復能處理的）

### 3. 修復

- 只改必要的地方，不做額外重構
- 如果原本有測試，確認測試更新涵蓋這個 bug 情境
- commit message：`fix: {問題描述}`

### 4. 驗證

- 重現步驟再跑一次，確認修好了
- 跑既有測試，確認沒有 regression
- 如果有 lint / type check，確認通過

### 5. 報告

告訴使用者：
- 根因是什麼
- 改了哪些檔案
- 怎麼驗證的

## 調查後修復流程（原因不明）

### 1. 蒐集線索

- 重現 bug，記錄完整的錯誤訊息和行為
- 如果無法重現，釐清觸發條件（環境、操作順序、資料狀態）
- 查 git log 看最近改了什麼，有沒有可疑的 commit

### 2. 追蹤根因

- 從症狀出發，往上游追蹤 code path
- 加 log / breakpoint 縮小範圍（二分法思路：先確認資料在哪一層開始不對）
- 查相關的 config、環境變數、外部依賴的狀態
- 如果是間歇性 bug，考慮 race condition、cache、timing 問題

### 3. 確認根因

找到根因後，向使用者報告：

```
## 調查結果

**症狀：** {使用者描述的問題}
**根因：** {實際出錯的原因}
**出錯位置：** {檔案:行號}
**影響範圍：** {哪些功能 / 情境受影響}
**建議修復方式：** {怎麼改}
```

等使用者確認方向後再繼續。如果根因明確且修復方式簡單，可以直接進入修復。

### 4. 規劃修復

根據複雜度決定做法：

- **簡單修復**（1-3 個檔案）→ 直接改，走快速修復流程的「修復 → 驗證」
- **複雜修復**（涉及架構、多檔案連動）→ 依序執行以下步驟，**每步完成後提醒使用者 `/clear` 再繼續**：
  1. `/speckit.plan` → 完成後提醒 `/clear`，再輸入 `/bug` 繼續
  2. `/speckit.tasks` → 完成後提醒 `/clear`，再輸入 `/bug` 繼續
  3. `/speckit.implement-opt` → 執行實作

### 5. 驗證

- 原始 bug 的重現步驟確認修好
- 跑測試確認沒有 regression
- 如果是複雜修復，額外檢查影響範圍內的相關功能

### 6. 報告

告訴使用者：
- 調查過程摘要（查了什麼、排除了什麼）
- 根因和修復方式
- 改了哪些檔案
- 驗證結果

---

## 通用原則

- **先理解再動手**：不要看到錯誤就猜，先讀 code 確認
- **最小改動**：只修 bug，不順便重構、不加功能
- **驗證是必須的**：每次修復都要驗證，不能只改 code 就結束
- **模糊就問**：不確定問題描述的意思、不確定修復方向，就問使用者
