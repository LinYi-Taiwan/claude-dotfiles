# Issue-to-Development Pipeline

Orchestrate the complete workflow from a GitHub issue to a merged PR, with two human approval gates.

## Pipeline Stages

### Stage 1: Issue Intake

1. Accept input: a GitHub issue URL, issue number + repo, or a request to scan for new issues.
2. If scanning for new issues:
   - Use the GitHub MCP to list open issues in the repo filtered by the `needs-triage` label.
   - Present the list and ask the user which to process, or process all automatically if instructed.
3. Fetch the full issue details using the GitHub MCP.
4. Extract: title, description, labels, author, linked PRs, comments.
5. Add label: `status:triaging`

### Stage 2: Multi-Angle Review

Invoke the **multi-angle-review** skill.

Each agent (PM, Tech Lead, UX) analyzes the issue and produces a structured assessment. Synthesize a conclusion and post it as a GitHub comment.

### Stage 3: Design Phase (if needed)

If the review determined UI changes are needed:

1. Invoke the **design-mockup** skill.
2. Use Figma MCP to create or reference design mockups.
3. Post the design link/screenshot as a GitHub comment.

If no UI changes are needed, skip to Stage 4.

### Stage 4: FE Feasibility Review

Invoke the **fe-feasibility** skill:

1. Check the existing codebase for reusable components.
2. Estimate effort and identify technical risks.
3. Post the feasibility report as a GitHub comment.

If rejected → flag for design revision and loop back to Stage 3.

---

### 🚦 APPROVAL GATE 1: Implementation Plan Approval

After the multi-angle review, design (if any), and feasibility report are complete, post a consolidated **approval request** comment on the issue:

```markdown
## 🚦 Approval Required — Implementation Plan

All review phases are complete. Summary:

### Review Conclusion

- **Recommendation**: [from review]
- **Priority**: [P0-P3]
- **Effort**: [S/M/L/XL]
- **Design Required**: [Yes/No]

### Proposed Implementation

[Tech lead's suggested approach + feasibility report highlights]

### Design Reference

[Link to Figma mockup, if applicable]

### Key Risks

1. [Risk 1]
2. [Risk 2]

---

**To approve**: comment `/approve` on this issue
**To request changes**: comment with your feedback

> Requires approval from at least {N} team member(s).
> Current approvals: 0/{N}

---

_🤖 Generated by issue-to-dev pipeline_
```

Add label: `status:awaiting-approval`
Remove label: `status:triaging`

**The pipeline stops here and waits for human approval.** Use the **check-approvals** skill to check if the issue has been approved.

---

### Stage 5: Development (after Approval Gate 1 passes)

Once approved, invoke the **development** skill:

1. Remove label `status:awaiting-approval`, add `status:in-development`
2. Create a feature branch from the issue.
3. Implement based on the review conclusions and design.
4. Run the code-reviewer agent for quality checks.
5. Create a PR linking back to the original issue.

---

### 🚦 APPROVAL GATE 2: PR Review Summary

After the PR is created, post a **PR summary comment** on the original issue AND as the PR description:

```markdown
## 🚀 PR Ready for Review

### PR: #<pr-number>

**Branch**: `feat/issue-<number>-<description>`

### What Changed

- [File-by-file or area-by-area summary of changes]

### How to Test

1. [Step-by-step testing instructions]

### Screenshots

[Before/after screenshots if UI changes]

### Checklist

- [x] All tests passing
- [x] Type checks clean
- [x] Linting clean
- [x] Code review by code-reviewer agent: [Approved/Changes requested]

### ⚠️ Things to Watch

- [Deployment considerations]
- [Migration steps if any]
- [Feature flags if any]
- [Rollback plan]

---

_🤖 Generated by issue-to-dev pipeline — please review and merge when ready._
```

Add label: `status:pending-merge`
Remove label: `status:in-development`

**The pipeline ends here.** The user/team reviews the PR and merges manually at their discretion.

---

## Manual Invocation

Invoke any individual stage by calling the corresponding skill directly:

- **multi-angle-review** — Run just the review phase
- **design-mockup** — Run just the design phase
- **fe-feasibility** — Run just the feasibility check
- **check-approvals** — Check if an issue has been approved
- **development** — Run just the development phase

## Label Convention

| Label                      | Meaning                               |
| -------------------------- | ------------------------------------- |
| `status:triaging`          | Pipeline is analyzing the issue       |
| `status:reviewed`          | Review complete                       |
| `status:awaiting-approval` | Waiting for human approval to proceed |
| `status:approved`          | Approved, ready for development       |
| `status:in-development`    | Currently being implemented           |
| `status:pending-merge`     | PR created, waiting for human merge   |

## Execution Notes

- Always confirm the repository and issue before starting.
- Post progress updates as GitHub comments at each stage transition.
- If any stage fails, stop and report the failure clearly.
- Never skip an approval gate — these are mandatory human checkpoints.
