# Development Phase

Implement the feature or fix based on the multi-angle review conclusions and design spec. Create a PR with automated code review and a comprehensive summary for human merge decision.

## Prerequisites

Before starting development, verify:

- The issue has the `status:approved` label (Approval Gate 1 passed).
- If invoked manually without prior approval, warn the user and ask for confirmation.

## Input

Accept:

- A GitHub issue (URL or number) with review and feasibility conclusions
- Specific implementation instructions
- Design spec or mockup reference

## Process

### 1. Preparation

1. Read the issue, all review comments, and the design/feasibility reports.
2. Identify the repository and default branch.
3. Update labels: remove `status:approved`, add `status:in-development`.
4. Create a feature branch: `git checkout -b feat/issue-<number>-<short-description>`
5. Plan the implementation based on:
   - The tech lead's suggested approach
   - The feasibility report's component mapping
   - The design spec's component breakdown

### 2. Frontend Implementation (React + TypeScript)

Follow this order:

1. **Types/Interfaces** — Define data models and prop types first.
2. **API layer** — Add API calls, hooks, or service functions.
3. **Components** — Build from smallest (atoms) to largest (pages).
   - Create new shared components in the design system directory.
   - Extend existing components where noted in the feasibility report.
4. **State management** — Add stores/context if needed.
5. **Routing** — Add new routes/pages if needed.
6. **Styling** — Follow existing patterns (CSS modules, Tailwind, styled-components, etc.).

### 3. Backend Implementation (if applicable)

If the issue requires backend changes:

1. **API endpoints** — Add new routes/controllers.
2. **Data models** — Update database schemas or ORM models.
3. **Business logic** — Implement service layer.
4. **Validation** — Add input validation and error handling.

### 4. Testing

1. Write unit tests for new components and functions.
2. Write integration tests for API endpoints.
3. Update existing tests if behavior changed.
4. Run the full test suite: `npm test` or equivalent.

### 5. Code Review

Invoke the **code-reviewer** agent to:

- Check code quality and adherence to project conventions.
- Identify potential bugs or performance issues.
- Verify test coverage.
- Suggest improvements.

Fix any critical issues identified by the reviewer.

### 6. Create Pull Request

1. Commit all changes with a meaningful message referencing the issue:
   ```
   feat(#<issue-number>): <short description>
   ```
2. Push the branch: `git push -u origin <branch-name>`
3. Create a PR using the GitHub MCP.

**PR description must include the full summary below** — this is Approval Gate 2:

```markdown
## 🚀 PR for Issue #<issue-number>: <title>

### What Changed

[File-by-file or area-by-area summary. Be specific — list every significant change.]

**Files changed**: <count>
**Lines added**: <count> | **Lines removed**: <count>

### Why

[Link back to the issue and the approved implementation plan]
Closes #<issue-number>

### How to Test

1. [Detailed step-by-step testing instructions]
2. [Include specific URLs, user actions, expected results]

### Screenshots

[Before/after screenshots if UI changes were made]

### ⚠️ Things to Watch

- **Deployment**: [Any special deployment steps needed?]
- **Migrations**: [Database or config migrations?]
- **Feature flags**: [Any flags to enable/disable?]
- **Breaking changes**: [Any breaking changes to APIs or contracts?]
- **Rollback plan**: [How to safely rollback if something goes wrong]
- **Performance**: [Any performance implications?]

### Checklist

- [ ] All tests passing
- [ ] Type checks clean (`npx tsc --noEmit`)
- [ ] Linting clean (`npm run lint`)
- [ ] Code review agent: [✅ Approved / ⚠️ Warnings]
- [ ] Matches approved design spec
- [ ] Accessibility checked

---

_🤖 Generated by issue-to-dev pipeline — review and merge at your discretion._
```

### 7. Post-Development

1. Update issue labels: remove `status:in-development`, add `status:pending-merge`.

2. Post a brief notification comment on the issue:

```markdown
## 🚀 PR Created: #<pr-number>

Development is complete. Please review the PR for:

- Change summary and impact analysis
- Testing instructions
- Deployment considerations

**Merge when ready** — no further automation needed.

---

_🤖 Generated by issue-to-dev pipeline_
```

**The pipeline stops here.** The human reviews the PR and merges manually.

## Guidelines

- Follow existing code conventions and patterns in the repository.
- Keep PRs focused — one issue per PR.
- Write descriptive commit messages.
- Don't introduce new dependencies without justification.
- Ensure all linting and type checks pass before creating the PR.
- The PR summary is critical — it's the team's decision-making reference for merge.
